0そもそもClaude Codeって何？

ふだんチャットで使っているClaude（Webアプリ）は、会話でいろんなことを相談できるAIです。
Claude Codeは、それの「実際に手を動かしてくれる版」です。

ふつうのClaude Claude Code
使う場所 ブラウザのチャット画面 ターミナル（黒い画面）
できること 質問に答える
コードを「表示」する		 実際にファイルを作る
コードを書いて保存する
アプリを動かすところまでやる
作業するのは 自分（コピペが必要） AIが自動でやってくれる
たとえるなら
ふつうのClaudeが「なんでも相談できる友達」だとしたら、Claude Codeは「隣に座ってくれるプログラミングの先輩」。
あなたのパソコンの中身を見ながら、一緒にコードを書いたり直したりしてくれます。

1安心して始める準備

まずは「失敗しても大丈夫」な状態を作ります。ここが一番大事です。

1-1. ターミナルを開いてみよう

ターミナルとは？
パソコンに文字で命令を出す画面のこと。
マウスでクリックする代わりに、文字を打って操作します。
見た目は黒い画面でちょっと怖そうですが、LINEでメッセージを送るのと同じ感覚です。

セットアップ済みの方は、Cursor の中のターミナルをそのまま使えます。
メニューの「Terminal」→「New Terminal」で開けます。

試しにこう打って Enter を押してみてください：

echo "はじめまして！"

パソコンが「はじめまして！」と返してくれます。これだけです。
何か打ち間違えてもパソコンが壊れることはありません。エラーが出ても「間違ってるよ」と教えてくれるだけなので、安心して試してOKです。

1-2. 「セーブポイント」の作り方を覚える

Git（ギット）とは？
ファイルの変更履歴を記録してくれる仕組み。
いつでも過去の状態に巻き戻せる「タイムマシン」のようなもの。

コミット ＝ 「ここまでの作業を記録する」こと。ゲームのセーブと同じ。

ゲームで難しいボス戦の前にセーブするように、作業前に「セーブポイント」を作る方法を覚えましょう。これで失敗しても元に戻せます。

# ① 練習用のフォルダを作って、そこに移動する
mkdir my-first-project
cd my-first-project

# ② Gitを使えるようにする（最初の1回だけ）
git init

# ③ 何かファイルを作ってみる
echo "私の最初のプロジェクト" > README.md

# ④ セーブポイントを作る（この2行はセットで覚える）
git add .
git commit -m "最初のセーブ"
これだけは覚えてください
Claude Codeで何か作業する「前」に、必ずセーブする癖をつけましょう。
うまくいかなかったら git checkout . で最後のセーブポイントに戻れます。

1-3. Claude Codeを起動してみよう

セットアップが済んでいれば、ターミナルで以下を打つだけで起動できます。

# 練習用フォルダに移動して起動
cd my-first-project
claude
画面が変わって、下のほうに > のような入力欄が出たら起動成功です！
ここに日本語で指示を打ち込めば、AIが作業を始めてくれます。

1-4. 「フォルダ」の考え方を知ろう

たとえるなら
フォルダは「引き出しのある棚」。
棚（フォルダ）の中にさらに小さい引き出し（フォルダ）があって、その中に書類（ファイル）が入っている。
物が整理されていれば必要なものがすぐ見つかる。コードもまったく同じです。
散らかった例 ❌
my-project/
├── aaa.html
├── test2.html
├── style.css
├── style2.css
├── old-backup.html
└── memo.txt

何がどこにあるか、わからない...

整理された例 ⭕
my-project/
├── index.html      ← トップページ
├── about.html      ← 自己紹介
├── css/
│   └── style.css   ← 見た目
├── images/
│   └── photo.jpg   ← 写真
└── CLAUDE.md       ← AI用メモ

ポイントは3つ：

  • 種類ごとに分ける。 写真は「images」、デザインは「css」のように同じ種類をまとめる
  • 名前でわかるようにする。 「aaa.html」ではなく「index.html」のように中身がわかる名前をつける
  • 深くしすぎない。 フォルダの中のフォルダの中の...と何段も深くなると迷子になる。3〜4段が目安
なぜこれが大事？
Claude Codeはフォルダの構造を見て「どこに何があるか」を判断します。
整理されていると正しいファイルを見つけてくれるし、散らかっていると間違ったファイルを触ってしまうことがあります。
人間もAIも、整理された部屋のほうが仕事しやすいのは同じです。

フォルダ構成に自信がなくても大丈夫。Claude Codeに「このプロジェクトのフォルダ構成を整理して」と頼めば、きれいに整理してくれます。

2基本の「型」を覚える（3つだけ）

3つだけ覚えれば、もう使えます。

型その① ─ 「メモ」を置いておく（CLAUDE.md）

プロジェクトのフォルダに CLAUDE.md というファイルを置くと、Claude Codeが最初に読んでくれます。
「このプロジェクトはこういうものだよ」と伝えるメモです。

たとえるなら
新しく入ったバイトさんに渡す「お店のルール表」のようなもの。

「何を書けばいいかわからない...」と思いますよね。大丈夫、Claude Codeに作ってもらえばいいんです。

私はカフェの予約サイトを作りたいと思ってます。
このプロジェクトに合ったCLAUDE.mdを作ってくれませんか？

やりたいことに合わせた頼み方の例：

やりたいこと頼み方
ブログを作りたい「個人ブログを作りたいので、CLAUDE.md を作って」
お店のHPを作りたい「パン屋のホームページを作りたいので、CLAUDE.md を作って」
業務ツールを作りたい「社内の在庫管理ツールを作りたいので、CLAUDE.md を作って」
まだ決まってない「プログラミングの練習がしたいので、初心者向けの CLAUDE.md を作って」
自分で一から書く必要はありません。やりたいことを伝えるだけで、専用のメモが出来上がります。
あとは使いながら「これも追加して」と育てていけばOKです。

型その② ─ 「どこの・何を・どうして」で頼む

Claude Codeへの頼み方にはコツがあります。
「どのファイルの」「何を」「どうしてほしい」の3つを伝えるだけで、精度がグッと上がります。

わかりやすい頼み方 ⭕
「index.htmlのタイトル部分を
『私のブログ』に変えて」
ふわっとした頼み方 ❌
「なんかいい感じにして」

頼み方の例文集：

やりたいこと頼み方の例
ファイルを読んでほしい「index.html を読んで、何が書いてあるか教えて」
修正してほしい「style.css の文字の色を、黒から紺色に変えて」
新しく作ってほしい「"お問い合わせ"ページを新しく作って。名前とメールの入力欄をつけて」
説明してほしい「app.js の10行目あたり、何をしてるか日本語で説明して」
エラーを直してほしい「このエラーが出たんだけど、原因を教えて直して」

型その③ ─ 「一個ずつ」頼む

一度にたくさん頼むと、結果がぐちゃぐちゃになりがち。
小さく頼んで、確認して、次に進む。この繰り返しが最強です。

小さいお願い
をする
→
結果を
確認する
→
OKなら
次のお願いへ
→
ダメなら
戻ってやり直す
一個ずつ ⭕
「まずヘッダーを作って」
→ 確認 →
「次に本文を作って」
まとめて ❌
「ヘッダーと本文とフッターと
お問い合わせフォームを
全部作って」

3しっかり使いこなすための基本

ここまでで、Claude Codeを動かすことはできるようになりました。
でも、「もっとちゃんと使いこなしたい」と思ったら、ここからが本番です。
むずかしい話はしません。でも、この章を読んでおくと、Claude Codeとの付き合い方がガラッと変わります。

3-1. MDファイル（Markdown）って何？

「見た目を整えられる、ただのテキストメモ」です。
Wordでもない、普通のメモ帳でもない、スマホのメモアプリで書いた文章をイメージしてください。

ファイル名の最後が .md になっているファイルが全部Markdownです。

💡 なぜClaude Codeが好きなの？
人間もAIも両方がスラスラ読めるから。見出しや箇条書きが # や - だけで書けて、どんなエディタでも開ける。

よく出てくる .md の種類

ファイル名 役割
CLAUDE.md Claude Code用のルール表（毎回自動で読まれる）
README.md プロジェクトの説明書（他の人が最初に読む）
docs/〇〇.md 詳しい仕様や手順書（必要なときだけ読む）

3-2. コンテキストって何？

「Claudeが今の会話で覚えていられる量」のことです。

🖥 机の広さをイメージしてください
会話が短いうちは机の上が空いていて快適。長くなると書類が机いっぱいに広がって作業しづらくなり、机からあふれた書類はClaudeが忘れていきます。

こんな症状が出たらコンテキストが重いサイン

  • 返事が遅くなってきた
  • さっき言ったことを忘れている
  • 急に的外れな返答が増えた

対処法

  • /clear … 机の上をまっさらにする（最初からやり直し）
  • /compact … 書類を要約して圧縮する（話の流れは引き継ぐ）

3-3. CLAUDE.md の書き方

CLAUDE.mdは、Claude Codeが毎回会話の最初に自動で読み込むルール表です。
ここに書いておけば、毎回同じ説明をしなくて済みます。

書いておくべき8つの項目

# 項目 例
①プロジェクトの目的「AI導入支援のLP」
②使っている技術Python / Next.js
③よく使うコマンドnpm run dev
④ファイル構造src/ はコード
⑤やってほしいこと「UIは白ベース」
⑥やってほしくないこと「.envをコミットしない」
⑦専門用語「王騎＝業務管理AI」
⑧安全ルール「pushは必ず確認」
📝 書き方のコツ
  • 命令形で書く：「〜してください」より「〜する」「〜禁止」の方が守られやすい
  • 箇条書き中心に：長い文章は読み飛ばされやすい
  • 重要ルールは上に：最初に書いたものほどよく守られる
  • 具体的に：「きれいに書いて」ではなく「関数は30行以内」
⚠️ 書かなくていいもの
  • 今日のタスク・一時的な予定
  • バージョン番号など変わりやすい情報
  • 「TypeScriptとは…」のような一般説明
  • 使われなくなった古いルール

3-4. CLAUDE.md は200行を超えないこと

なぜ200行ルールがあるのか

  • CLAUDE.mdは毎回の会話で読み込まれる
  • 長すぎるとコンテキスト（机）を圧迫する
  • 長いほど、1つ1つのルールが薄まって守られにくくなる
  • 一部のツールでは201行目以降が自動で切り捨てられることもある

目安

  • 🟢 100行前後 … ベスト
  • 🟡 150行 … そろそろ見直し時期
  • 🔴 200行手前 … 分割を検討

超えそうなときの対処法

  1. 似たルールをまとめる（同じ話が何回も出ていないか確認）
  2. 詳細は別ファイルへ（「詳しくは docs/xxx.md を見て」と書くだけ）
  3. サブフォルダに分散（次項のMD階層を参照）
  4. 古いルールを削除（使っていないルールは容赦なく消す）

3-5. MDファイルの階層設計

基本の考え方は「全部ひとつのCLAUDE.mdに詰め込まない」
フォルダごとに分けて、必要なときだけ読まれる仕組みを作るのがコツです。

階層の例

my-project/
├── CLAUDE.md              ← 全体のルール（100行以内）
├── docs/
│   ├── architecture.md    ← システムの全体像
│   ├── api-spec.md        ← API仕様の詳細
│   └── deployment.md      ← デプロイ手順
├── backend/
│   └── CLAUDE.md          ← backend専用ルール
└── frontend/
    └── CLAUDE.md          ← frontend専用ルール

役割分担（3階層で考える）

場所 役割 たとえると
ルートの CLAUDE.md 全体の羅針盤・最上位ルール 会社の就業規則
サブフォルダの CLAUDE.md その場所だけのルール 部署のマニュアル
docs/ の .md 詳しい資料（必要なときに見る） 社内Wiki
📝 ポイント
  • サブフォルダのCLAUDE.mdは、そのフォルダで作業中のときだけ自動で読まれる
  • ルートは薄く、サブは具体的に
  • 重複するルールは書かない（矛盾の元）
⚠️ やってはいけない階層
  • ルートに全部詰め込んで500行越え
  • 同じルールをあちこちにコピペ
  • 階層が4段・5段と深すぎる

3-6. スキルって何？

「この作業はこの手順でやって」を、あらかじめ登録しておく機能です。

たとえば「ブログ書いて」と言っただけで、
テーマ選定 → 構成案 → 本文 → 画像生成 → WordPress投稿 まで自動で進む。
そんな"おまとめボタン"のようなものです。

使うとこうなる

  • 一言で複雑な作業が完結する
  • 同じ作業の品質がブレない
  • あとから自分で作って増やせる

スキルが向いている作業

  • 毎回同じ手順を踏む作業（ブログ投稿・請求書作成など）
  • 複数のステップをまたぐ作業
  • 決まったルールに沿って書く作業

3-7. よく使うコマンド

スラッシュコマンド（/ で始まるやつ）

コマンド 何をする？ いつ使う？
/helpヘルプを表示困ったとき
/clear会話をリセット話が長くなってきたら
/compact会話を要約圧縮続けたいけど重いとき
/initCLAUDE.mdを自動生成プロジェクト始めに
/modelAIモデルを切り替え軽い作業に切り替えたい
/cost料金の確認いくら使ったか知りたい

キー操作

キー 動き
Esc今の作業を止める
Shift + Tabモードを切り替え
Ctrl + C強制終了
↑キー前に送った文を呼び出す

3-8. モードの違い

Claude Codeには3つの動き方があります。Shift + Tab で切り替え。

モード どう動く？ こんなとき
通常モード1つずつ確認しながら進む初心者はまずこれ
プランモード計画だけ立てて実行しない大きい作業の前に
自動承認モード確認せずどんどん進む慣れてきたら

最初は通常モードでOK。慣れたら使い分けるイメージです。

3-9. やらかし防止ルール

Claude Codeは強力なので、やらかすとファイルが消えたりすることもあります。これだけは守ってください。

⚠️ 3つの鉄則
  1. 作業前に必ずコミット（セーブポイントを作ってから頼む）
  2. 「全部消して」のような曖昧な指示はNG（何が消えるか分からない）
  3. 本番環境・公開サービスに直接つながない（練習は必ず自分のフォルダで）

困ったら

  • 変な動きをしたらすぐ Esc で止める
  • コミットしてあれば git reset で戻せる
  • 何が起きてるか分からなくなったら、いったん /clear して冷静に

4困ったときの対処法

困ったこと 解決法
エラーが出た そのエラーメッセージをそのままClaude Codeに見せる。
「このエラーが出たんだけど、どうすればいい？」でOK。
思ったのと違うものができた 「これじゃなくて、こういう感じにしてほしい」と具体的に伝え直す。
または git checkout . で元に戻してやり直す。
何を頼めばいいかわからない 「今あるファイルを見て、何ができるか教えて」と聞いてみる。
Claude Codeが提案してくれます。
全部ぐちゃぐちゃになった git checkout . で最後のセーブポイントに戻れる。
だからこそ、こまめなセーブが大事！
Claude Codeが止まった Ctrl + C で一度終了してから claude で再起動。
何もわからなくなった Claude Codeに「今のやつ、もっとわかりやすく説明して」と聞けばOK！
何度聞いても嫌がらないので、遠慮なく聞いてください。

5用語じてん

出てきた言葉をやさしく説明しています。忘れたらここに戻ってきてください。

ターミナル
パソコンに文字で命令する画面。LINEでメッセージを送る感覚で使える。
Git（ギット）
ファイルの変更履歴を記録する仕組み。タイムマシンのように、過去に戻れる。
コミット
「ここまでの作業を記録！」という操作。ゲームのセーブポイント。
npm（エヌピーエム）
ツールをインストールするお店。App Storeのターミナル版。
Node.js（ノードジェイエス）
npmを動かすために必要な土台。一度入れたらあとは意識しなくてOK。
CLAUDE.md
Claude Code用の「お店のルール表」。プロジェクトのルールを書いておくメモ。
HTML / CSS
Webページを作るための言語。HTMLが「骨組み」、CSSが「見た目の装飾」。
リファクタリング
コードの動きは変えずに、書き方だけを整理整頓すること。部屋の模様替えのイメージ。
フォルダ / ディレクトリ
同じもの。ファイルを分類して入れる「箱」のこと。

あせらず、一歩ずつ。

わからないことがあったら、Claude Codeに
「今のやつ、もっとわかりやすく説明して」と聞けばOK！
